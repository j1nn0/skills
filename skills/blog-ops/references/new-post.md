# 新規記事のセットアップ

記事を書き始める前の下準備。ここで規約に合わせておくと、公開前チェックで手戻りしない。

## 手順

### 1. スラッグを決める

小文字・ハイフン区切りの英語スラッグにする(例: `my-keyboard-review`)。
スラッグはそのまま URL(`/posts/<slug>/`)と画像ディレクトリ名になるので、公開後の変更はリダイレクトが必要になる。最初に内容を表す簡潔な名前を決めること。

既存記事と重複しないか `ls content/posts/` で確認する。

### 2. 記事ファイルを作る

`content/posts/<slug>.md` を以下の front matter で作成する。

```yaml
---
title: "記事タイトル"
slug: <slug>
date: YYYY-MM-DD
summary: "記事の結論と扱う範囲を一文で示す"
tags:
  - broad_theme
  - main_subject
draft: true
---
```

- `date` は作成日。公開時にずれていたら更新する。
- `draft: true` で始める。公開判断が出るまで false にしない。
- タイトルの付け方は blog-writing-guide-ja に従う。

### 3. summary / description を書く

`summary` は、記事一覧・OGP・検索結果で本文より先に読まれる短い説明である。本文の冒頭を切り出さず、結論と扱う範囲を1〜2文で書く。

- 日本語では 60〜120字を目安にする。媒体や表示幅によって途中で省略されるため、重要な結論は前半に置く。
- タイトルだけでは分からない対象、結果、判断材料を足す。「〜について解説します」のような予告や、本文にない約束は書かない。
- タイトルと続けて読んでも不自然にならず、記事単体でも意味が通る文にする。
- サイトが `description` を別に持つなら、同じ基準で書き分ける。どのフィールドを OGP や検索メタデータに使うかは対象リポジトリの設定を確認する。

### 4. タグを選ぶ

タグは読者向けの横断ナビゲーションであって、記事に出てくる技術の索引ではない。

1. まず既存タグを確認する: `ls content/tags/`
2. 3〜5個を選ぶ。広いテーマを先、記事の主題であるプロダクト・技術を後に並べる。
3. 表記は小文字+アンダースコア(`ai_agents`、`cloudflare_pages`)。ハイフン・スペースは使わない。
4. 既存タグで足りるなら新タグを作らない。新タグが必要なら `references/tags.md` の手順で追加する(タグページの作成が必要)。

記事の主題でない技術に言及しただけならタグにしない。迷ったら付けない側に倒す。

### 5. 画像の置き場所を用意する

画像を使う記事なら `static/images/<slug>/` を作り、そこに置く。
記事からの参照は `/images/<slug>/ファイル名` で、alt テキストは画面の内容を日本語で説明する(例: `![設定画面でプラグインを有効化している様子](/images/my-post/01-settings.webp)`)。

フォーマットはスクリーンショットなら webp か png。既存記事に合わせる。

### 6. 動作確認

`hugo server -D` でドラフトを含めて表示し、タイトル・日付・タグ・URL が意図どおりか確認する。
