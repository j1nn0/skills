# 作業規約

Hugo ブログのリポジトリで作業するときの既定の規約。リポジトリの AGENTS.md が別の定めをしていればそちらが優先。

## 構成(Hugo + PaperMod の既定)

| 場所 | 内容 |
|------|------|
| `hugo.yaml`(または `config.toml` 等) | サイト設定の正本(baseURL、OGP、言語設定など) |
| `content/posts/` | 記事(Markdown、小文字ハイフン区切りのスラッグ) |
| `content/tags/<内部値>/_index.md` | タグの表示名定義 |
| `layouts/partials/` | テーマオーバーライド(`extend_head.html`、`extend_footer.html` など) |
| `assets/css/extended/` | カスタム CSS(PaperMod の拡張ポイント) |
| `static/images/` | OG 画像などそのままコピーされる静的ファイル |
| `themes/<テーマ名>` | 上流テーマ(通常はサブモジュール) |

テーマ本体(`themes/` 配下)は直接編集しない。サイト側のオーバーライド(`layouts/`、`assets/`)で実現できるならそちらを使う。意図的にテーマを更新するときだけ例外。

## ビルド・開発コマンド

- `hugo server -D` — ドラフト込みでローカル開発サーバを起動
- `hugo` — `public/` に本番ビルド
- `hugo --minify` — リリース相当の最小化ビルド(公開前チェックで使う)
- `hugo --cleanDestinationDir` — ビルド時に古い生成物を削除

Hugo サイトに `package.json` のワークフローがあるとは限らない。`npm test` や `npm run build` を確認せずに前提にしない。

## コーディングスタイル

- YAML・HTML・CSS はインデント2スペース(リポジトリの既存スタイルに合わせる)。
- front matter は簡潔で妥当な YAML に保つ。
- コンテンツのファイル名は小文字ハイフン区切り(例: `content/posts/my-post.md`)。
- カスタム CSS はスコープを絞って書く。小さなオーバーライドで足りるならテーマの広範な書き換えをしない。

## スコープ制御

ブログは著者の声で書かれた資産であり、URL は公開後の契約である。次を守る。

- 依頼に必要な変更だけを行う。
- 執筆スキルを満たすためだけに無関係な記事を書き直さない。
- コンテンツのファイル名・スラッグ・タグ・タクソノミー・生成 URL を、影響の報告なしにリネームしない。
- 著者の事実の主張、個人的な経験、意見、不確かさの表明を保存する。
- テスト、結果、動機、使用歴、失敗、ツール選定の理由をでっち上げない。
- 推測を確定事実に変えない。
- 時間経過で変わりうる事実の記述を更新するときは、一次情報で確認する。
- 依頼されたファイルの外で広範な整形変更をしない。

## コミット規約

英語の Conventional Commits 形式。件名は72文字未満。

## セキュリティ・設定

- ローカルのシークレット、デプロイトークン、生成された認証情報をコミットしない。
- サイト設定ファイルが正本。`baseURL`、Open Graph、Twitter カード、言語設定を編集するときは本番 URL を注意深く確認する。
- `public/` は意図的に追跡していない限りコミットしない。
