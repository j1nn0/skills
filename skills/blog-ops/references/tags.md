# タグ運用

タグは Hugo の term URL(`/tags/<内部値>/`)を生成し、PaperMod の SNS シェアにも使われる。
表記を間違えると URL が壊れたりシェア用ハッシュタグが崩れたりするので、変更は必ずこの手順で行う。

## タグの二層構造

| 層 | 場所 | 例 | 用途 |
|----|------|----|----|
| 内部値 | 記事 front matter の `tags:` | `development_environment` | URL 生成、SNS シェア(X のハッシュタグ) |
| 表示名 | `content/tags/<内部値>/_index.md` の `title:` | `開発環境` | タグページ見出し、記事フッター、タグ一覧 |

内部値の規約: 小文字+アンダースコア(`ai_agents`)。ハイフン・スペース禁止。
PaperMod がホワイトスペースを除去してから SNS の URL に渡すため、この形式でシェア値が安定する。

## タグを追加する

1. 既存タグで代替できないか `ls content/tags/` で確認する。タグは記事横断のナビゲーションなので、1記事にしか付かないタグは基本作らない。
2. 記事の front matter に内部値を追加する。
3. タグページを作る:
   ```yaml
   # content/tags/<内部値>/_index.md
   ---
   title: 表示名
   ---
   ```
4. `hugo server -D` でタグページ・記事フッター・タグ一覧の表示名を確認する。

## タグを改名する

改名は「内部値の変更」と「表示名の変更」で影響がまったく違う。まずどちらの話か確定させる。

**表示名だけ変える**(URL 不変・安全):
`content/tags/<内部値>/_index.md` の `title:` を変えるだけ。リダイレクト不要。

**内部値を変える**(URL が変わる・慎重に):

1. 変わる term URL を新旧で列挙し、リダイレクト方針をユーザーに確認する。勝手に進めない。
2. 全記事の front matter で旧値→新値に置き換える(`/usr/bin/grep -rl '旧値' content/posts/` で対象を洗い出す)。
3. `content/tags/` のディレクトリを旧値→新値にリネームする。
4. 301 リダイレクトを追加する。方式はホスティング依存なので AGENTS.md で確認する(Cloudflare Pages なら `static/_redirects`):
   ```
   /tags/旧値/ /tags/新値/ 301
   ```
   日本語タグだった場合は URL エンコードした形で書く(例: `/tags/%E9%96%8B%E7%99%BA/ /tags/development/ 301`)。
5. check_posts.py で孤児・重複がないことを確認する。

## タグを削除する

1. 対象記事の front matter から外し、`content/tags/<内部値>/` を削除する。
2. リダイレクトは明確な後継タグがあるときだけ張る。後継がなければ張らない。代表的な関連記事1本に誘導する(`/tags/旧値/ /posts/関連記事/ 301`)選択肢もあるが、これはユーザーに提案して判断を仰ぐ。

## 変更後の確認

- 記事フッターのラベル、タグ一覧ページのラベル、タグページの URL、SNS シェア値の4点を確認する。
- SNS シェアが内部値(アンダースコア区切り)を使っていること。表示名が渡っていたら間違い。
- check_posts.py で規約・孤児・重複を検査する。
