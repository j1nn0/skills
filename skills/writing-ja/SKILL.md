# writing-ja — 日本語記事の執筆・推敲規範

3つのスキル（japanese-tech-writing、human-writing-ja、humanizer-ja）を統合し、規範同士の衝突は j1nn0.com の公開記事（2026-06〜07）の実文体で裁定した。
このファイルだけで完結する。他のスキルを読み込む必要はない。

主対象は技術ブログ記事。メール・報告書は末尾の「ですます調の文書への差分」を重ねて適用する。

## 衝突の裁定

元スキル間で食い違っていた点は、実記事の用法を根拠に次の通り決めた。

| 論点 | 元スキルの対立 | 記事の実態 → 裁定 |
| --- | --- | --- |
| 太字 | 用語初出は太字 ↔ 全面禁止 | 本文で不使用 → 禁止 |
| コロン | 定義リストに全角コロン可 ↔ 禁止 | 不使用（■と表で代替） → 禁止 |
| 改行 | 一文一行 ↔ 文脈で判断 | 一文一行 → 一文一行 |
| 声・主観 | 入れない ↔ 体験を入れる | 体験と所感が記事の核 → 入れる |
| 見出し | オチを言わない ↔ 規定なし | 結論を言い切る見出しが多数 → 言い切ってよい |
| 脚注 | `[^ラベル]` を使う ↔ 規定なし | 不使用 → 使わない |
| 断定と推量 | 事実は断定しろ ↔ 不確実性を保て | 正直な留保を残している → 確定したことだけ断定 |
| 中黒（・）の並列 | 禁止 ↔ 許容 | 不使用 → 禁止（「と」「や」でつなぐ） |
| 読者への呼びかけ | 避ける ↔ 規定なし | 案内で使用 → 案内と結びに限り可 |

## 書式

- だ/である調。体言止めと軽い口語を混ぜてよい（「設定はこんな感じ。」「速くて安い。」）。
- 一文ごとに改行する。段落の区切りは空行。1段落は240字以内。
- 太字（`**`）を使わない。本文、箇条書き、引用、見出し、ラベルのすべてで禁止。
- コロン（`:` `：`）で見出し・文・ラベルを終えない。ラベルは■、注釈は※で書く。
- 全角ダッシュ・emダッシュ（——、—）を使わない。括弧か読点、または二文に分ける。
- 並列の中黒（・）を使わない。「と」「や」でつなぐ。固有名詞の内部は例外。
- 絵文字と感嘆符を使わない。所感は短文とリズムで出す。
- コード要素、パス、コマンドはバッククォートで囲む。
- 設定値・比較・割り当ては表にする。文章で並べない。手順の流れは番号付きリスト。
- 説明文中の箇条書きは連続3項目まで。チェックリスト、問題点の列挙、参照リストは例外。
- front matter の tags はアンダースコア区切り（`machine_learning`）。ハイフンはSNSのハッシュタグ認識を壊す。
- 文書を更新したとき、変更履歴・更新日・移行メモ・差分説明を本文に残さない。ゼロから書いたのと同じ仕上がりにする。

## タイトルと見出し

- タイトルは変化・成果を言い切る。「〜した話」「〜している話」型が定番。数値や固有名詞を入れてよい。
  - 実例「AIサブスクを3つ使い分けてレート制限に悩まなくなった話」
  - 実例「ターミナル起動を1秒も遅くせず、Homebrewを裏で安全に自動更新する」
- 「はじめに」「おわりに」「まとめ」の見出しを使わない。
- 見出しは結論を言い切るセンテンス型でよい。
  - 実例「v1 では Orchestrator が全部やってしまい、プリセットが活きなかった」
  - 実例「RAGにしないと決めた」
- 「〜における課題」「〜の概要」のような論文調の見出しは使わない。
- 結びの節は「使ってみて」「実際の動き」のような体験を指す句にする。
- 300字以下のセクションに小見出しを立てない。

## 記事の構成

記事は次の型で組む。

1. 課題ときっかけ。自分の体験・決断から入る。冒頭3〜4文で「何に困って、何をしたか」まで言い切る
2. 解決策の全体像。フローの番号付きリスト、構成の表、図で見せる
3. 各論。設定、コード、そして判断の理由
4. 使ってみての変化。導入前後の対比と短い所感

- 冒頭を「近年〜」「〜が注目されています」「ここでは〜について解説します」で始めない。いきなり自分の状況から入る。
  - 実例「OpenCode を使い始めてすぐに気づいたのが、レート制限の問題だ。」
- 結びは導入前後の対比を書き、最後は短い所感で締める。「〜が期待されます」で終えない。
  - 実例「実際にこの運用を始めてから、朝のターミナル起動時のもっさり感がなくなり、快適になった。」
  - 実例「でも圏論をC#で読めるノートは、自分でしか育てられなかった。」
- コードブロックの直前に導入の一文を置く。「設定はこんな感じ。」「定義全体を載せておく。」
- スクリーンショットの alt テキストは日本語で画面の状況を書く。

## 段落と論証

- 一つの段落には一つのトピックだけを置く。段落の最初の文で、その段落が何の話かわかるようにする。
- 論証は一方向に進める。結論を出してから反論を処理して結論を言い直す構成にしない。
- 因果を主張するときは、その機構（なぜそうなるのか）を一文で示す。「AだとBになる」とだけ書かない。
- 選定・割り当てには理由を書く。何かを選んだ記述には「なぜそうしたか」を添える。
  - 実例「DeepSeek V4 は Ollama Cloud より OpenCode Go のほうがコスパがいいので、DeepSeek V4 を使うエージェントは OpenCode Go に寄せる。」
- 異なるものを「同じ」とまとめない。区別すべき対象は腑分けして書く。複数の要因がある事象を単一の原因に還元しない。
- 検出・保証・解決を「必ず」できるかのように書かない。条件付きで正確に述べる（「〜しやすい」「〜できることが多い」）。
- 主張は、挙げた例が実際にその全体を支えているかを確認する。例が一部しか支えないなら主張の範囲を狭める。
- 「後で扱う」と前方に逃がした論点は、本当に回収する。前方参照は段落末・節末に置き、論証の途中に挟まない。

## 断定と推量

- 本文内の根拠で確定していることは言い切る。根拠なく弱めた「〜と言えるでしょう」「〜かもしれません」は断定に直す。
- ただし機械的に断定へ変えない。未確認の可能性、推定、読者が抱きそうな疑念を表す文は、不確実性を保ったまま整える。
- 確認していないことを、確認したかのように滑らかに書かない。試していないなら「まだ試してない」と書く。

## 声

- 一人称は「自分」。「筆者」「私」は使わない。
- 体験か意見を最低1箇所入れる。導入した理由、使ってみた感触、気に入っている点。
- 正直な留保を隠さない。できていないこと、やらなかった理由をそのまま書く。
  - 実例「記事ごとに変える仕組みはまだ用意してない（そのうちやる）。」
  - 実例「調べるのが面倒になってやめた。」
  - 実例「正直あまり意味がなかった。」
- 読者への案内は「〜を参照してほしい」。詳細を公式ドキュメントに任せる部分は潔く任せる。
- 語尾とリズムを揃えすぎない。同じ語尾を3回以上続けない。長い文の後に短い文を置く。

## 具体性

- 「さまざまな」「多くの」「あらゆる」は具体的な数字・固有名詞・事例に置き換える。
- 出典は具体的に書く。「業界の専門家によると」ではなく資料名と発行元。出せないなら自分の意見として書く。
- 「重要です」「必要です」で文を終えない。理由や機構を一文添える。
- カタカナ語が3語以上連続したら疑う。和語で言えるなら和語を使う。

## AI臭の除去

以下は論点を増やさず「ちゃんと書いている感」だけを付ける。見つけたら消すか、具体的な内容に置き換える。

| 分類 | 禁止表現 |
| --- | --- |
| 予告と総括 | 「重要なのは〜である」「本記事では〜を解説します」「まとめると」（言い換えだけのとき） |
| 定型評価語 | 「浮き彫りにしており」「今後の展開が注目されます」「注目に値する」「重要な示唆を与えている」 |
| 空虚な形容 | 「不可欠」「核心的」「鍵となる」「画期的」「多角的」「包括的」「極めて」「非常に」 |
| 空虚な動詞 | 「掘り下げる」「深掘りする」「言語化する」 |
| 締めのテンプレ | 「〜が期待されます」「〜と言えるでしょう」「〜ではないでしょうか」 |
| 説教くさい前置き | 「ここで重要なのは」「注意すべき点として」「〜を理解しておく必要があります」 |
| 追従 | 「素晴らしいご質問ですね」「おっしゃる通り」 |

- 「〜することができます」は「〜できます」に必ず短縮する。
- データや事実の後に意義を付け足さない。「〜を示しており」「〜を物語っています」は文ごと削除する。
- 「〜ではない。〜だ。」の対句を2回以上使わない。
- 何でも3つにまとめない。4つあるなら4つ書く。1つなら1つでいい。
- 効果・意義を表す紋切り述語（「実現した」「堅牢な」「大幅に」「恩恵を受けられる」）を連打しない。効果は1回だけ、具体的に書く。

■問題（過去記事に実在したAI臭）

```
高速な静的解析とフォーマットにより、CI やローカルでのコード品質チェックがほぼ一瞬で完了するようになり、コード変更時のフィードバックサイクルを大幅に短縮できた。
型安全性の確保によって、複雑な入力追跡ロジックに明確な型定義が与えられ、安全に機能拡張ができる堅牢な基盤が整った。
```

■改善

```
oxlint はプロジェクト全体のチェックが1秒かからない。lint 待ちがなくなった。
TypeScript 化で入力追跡ロジックに型が付いたので、壊す不安なくいじれるようになった。
```

## 冗長の排除

- 同じ主張を言い換えて繰り返さない。1つの主張は1回だけ書く。
- 場面や事実を述べた直後に要約し直さない。意味づけの一文だけを置く。
- 接続や評価のためだけの文を置かない。
- 「それ」「その」「これ」「この」を減らす。省略するか具体的な名詞に置き換える。
- 同じ主語が続くときは最初だけ書く。
- 「さらに」「また」「加えて」を連打しない。半分は削除して、文の順序で論理を伝える。段落頭で論理関係を示す接続（「かといって」「そこで」「ただし」）は冗長と見なさない。
- 読者が自力で補える中間段階の説明は書かない。
- 後で参照する必要のない固有名（ファイル名、関数名）を出さない。一般的な言い方で済ませる。

## ですます調の文書への差分

メール、報告書、企画書などのビジネス文書では、上記に次の差分を重ねる。

- ですます調にする。議事録は体言止め。
- 体験・感想・意見を入れない。事実のみ書く（「声」の節は適用しない）。
- 一人称を使わない。
- 見出しは対象を指す句にする（センテンス型は使わない）。
- 改行は文脈で判断する。話題が変わるときだけ空行を入れる（一文一行は適用しない）。
- 「です」が3回続いたら、4回目は「でした」「になります」に変える。

## 検証手順

修正完了後、Grepで以下を検索する。検出されたら修正し、0件になるまで繰り返す。目視だけでは見落とす。

| 検索パターン | 期待結果 |
| --- | --- |
| `——` `—` | 0件 |
| `\*\*` | 0件（引用内・リスト内も含む） |
| `[:：]\s*$` | 0件（コードブロック内は除く） |
| `^#{1,6}.*[:：]` | 0件 |
| `することができ` | 0件 |
| `浮き彫り\|展開が注目\|言えるでしょう\|深掘り\|ではないでしょうか` | 0件 |
| `筆者\|私は` | 0件（記事の場合） |
